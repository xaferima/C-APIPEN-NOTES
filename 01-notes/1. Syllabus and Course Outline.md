# ­¤ōÜ Comprehensive Syllabus and API Audit Course Outline - C-APIPen

This document is your master study guide and interactive checklist. It covers in hyper-detailed form each attack family that can occur in an API environment (REST, SOAP, GraphQL), detailing concepts, discovery methodologies, exploitation variants, success signatures, and key tools.

---

## ­¤öŹ 1. Reconnaissance and Attack Surface Discovery
*Mapping the API is 90% of success. If you don't know an endpoint, you can't attack it.*

- ### 1.1 API Documentation Exposure (Swagger, OpenAPI, RAML, WSDL)
  - [ ] **Description:** Locating specification files that detail routes, parameters, data schemas, and authentication schemas.
  - [ ] **Variants to Test:**
    - Brute force common endpoints: `/swagger.json`, `/swagger-ui.html`, `/api-docs`, `/v1/api-docs`, `/openapi.json`, `/api/v2/swagger.json`, `/swagger/v1/swagger.json`.
    - Search for HTML comments or exposed JS files in the frontend that point to these routes.
  - [ ] **Success Signals:** HTTP `200 OK` responses returning structured JSON schemas or the interactive Swagger UI viewer.
  - [ ] **Tools:** Burp Suite Extension *Swagger Parser*, Postman (import definition), `ffuf`.

- ### 1.2 Enumeration and Fuzzing of Hidden Endpoints
  - [ ] **Description:** Identifying API resources and controllers not listed in public documentation (debug endpoints, legacy versions, administrative functions).
  - [ ] **Variants to Test:**
    - Fuzzing routes with specialized dictionaries: `/api/v1/FUZZ`, `/api/FUZZ/users`.
    - Fuzzing non-conventional HTTP verbs on existing endpoints (e.g., changing `GET /users` to `POST`, `PUT`, `DELETE`, or `PATCH`).
    - Fuzzing versions in the URI: changing `/v2/` to `/v1/`, `/v0/`, `/v3/`, `/beta/`, `/dev/`.
  - [ ] **Success Signals:** HTTP status codes `401 Unauthorized` or `403 Forbidden` on administrative routes (confirming their existence) instead of `404 Not Found`.
  - [ ] **Tools:** `ffuf`, `kiterunner` (contextual scanning), `wfuzz`.

- ### 1.3 Discovery of Hidden Parameters
  - [ ] **Description:** Identifying internal variables accepted by the API that allow changing backend behavior (e.g., `?debug=true`, `?admin=1`, `?dest=...`).
  - [ ] **Variants to Test:**
    - Fuzzing parameters in URLs (GET) and request bodies (POST/JSON).
    - Injection of common control parameters (`X-HTTP-Method-Override`, `_method`).
  - [ ] **Success Signals:** Variation in HTTP response length (`Length`), detailed error codes, or debug traces appearing in the response.
  - [ ] **Tools:** `arjun` (intelligent fuzzing), `ffuf` (with Burp parameter dictionary).

## ­¤öæ 2. Authentication Mechanism Failures
*Failures that allow bypassing login, impersonating identities, or hijacking sessions.*

- ### 2.1 Brute Force and Password Spraying
  - [ ] **Description:** Automated attacks against the login endpoint using common password lists against one or multiple accounts.
  - [ ] **Variants to Test:**
    - Default credentials on system accounts (`admin:admin`, `guest:guest`, `root:root`).
    - Password spraying with weak seasonal passwords (e.g., `Spring2026!`) against a list of previously enumerated valid users.
  - [ ] **Success Signals:** HTTP `200 OK` or `302 Found` with delivery of valid session tokens.
  - [ ] **Tools:** `ffuf` (sending JSON payloads), Burp Intruder.

- ### 2.2 Bypassing Account Lockout and Rate Limiting on Login
  - [ ] **Description:** Circumventing policies designed to lock user accounts or slow down attacker requests after failed attempts.
  - [ ] **Variants to Test:**
    - Injection of network headers to simulate alternative origin: `X-Forwarded-For: 127.0.0.1`, `X-Real-IP: 10.0.0.X`, `CF-Connecting-IP`.
    - Altering insignificant parameters in the request (e.g., email capitalization, adding whitespace).
    - Abusing alternate endpoints that authenticate indirectly (e.g., `/api/v1/auth/refresh`, `/api/v2/login/mobile`).
  - [ ] **Success Signals:** The API continues processing requests with HTTP `200` or `401` instead of returning `429 Too Many Requests`.
  - [ ] **Tools:** Burp Suite.

- ### 2.3 Password Reset Flow Hijacking
  - [ ] **Description:** Abusing password reset flow logic to force generation of recovery links or OTP codes to attacker domains or arbitrary accounts.
  - [ ] **Variants to Test:**
    - **Host Header Injection:** Changing the `Host: target.com` header to `Host: attacker.com` to send the recovery link with the attacker's domain.
    - **HTTP Parameter Pollution (HPP):** Sending duplicate email parameters: `{"email": "victim@target.com", "email": "attacker@evil.com"}` or `email=victim@target.com&email=attacker@evil.com`.
    - **OTP Brute Force:** If the reset token is a short PIN (4-6 digits), automate testing without rate-limiting.
  - [ ] **Success Signals:** Receipt of HTTP requests on your controlled server with victim's reset tokens, or successful modification of victim's password.
  - [ ] **Tools:** Burp Collaborator / Netcat, `ffuf` for PIN brute force.

- ### 2.4 OTP/MFA Abuse (Multi-Factor Authentication Bypass)
  - [ ] **Description:** Circumventing the second-factor validation screen or API to access authenticated accounts.
  - [ ] **Variants to Test:**
    - Direct access to dashboard bypassing MFA call (/dashboard instead of /verify-mfa).
    - Race conditions in code validation.
    - Reuse of previously expired or already-used OTP codes.
    - Manipulate server responses (Response Manipulation): change `{"mfa_required": true}` to `{"mfa_required": false}` or change a `401` status code to `200 OK` in Burp proxy.
  - [ ] **Success Signals:** Access to protected environment without entering a valid OTP code.
  - [ ] **Tools:** Burp Proxy / Match and Replace.

## ­¤øĪ’ĖÅ 3. JSON Web Tokens (JWT) and Session Management
*Manipulation of mechanisms for storing and validating user state.*

- ### 3.1 Exploitation of "none" Algorithm (alg=none)
  - [ ] **Description:** Force server to accept a modified JWT token whose signature has been intentionally disabled.
  - [ ] **Variants to Test:**
    - Change JWT header to `{"alg": "none"}` or variations like `None`, `NONE`, `nOnE`.
    - Remove JWT signature (maintaining initial two dots, e.g., `eyJhbGciOiJub25lIn0.eyJ1c2VyIjoiYWRtaW4ifQ.`).
  - [ ] **Success Signals:** Server processes the request and recognizes the modified user (e.g., returning administrative information).
  - [ ] **Tools:** `jwt_tool`, Burp JWT Editor.

- ### 3.2 Algorithm Confusion (Key Confusion / RS256 vs HS256)
  - [ ] **Description:** Trick server into validating an asymmetric signature (RS256) using a symmetric algorithm (HS256) signed with the server's own public key.
  - [ ] **Variants to Test:**
    - Obtain server's public key (e.g., from `/jwks.json`, `/well-known/jwks.json`, or TLS metadata).
    - Convert public key format.
    - Change JWT algorithm to `HS256` and sign token using public key as symmetric signature secret.
  - [ ] **Success Signals:** Server validates signature and grants access.
  - [ ] **Tools:** `jwt_tool`, PyJWT locally.

- ### 3.3 "kid" Parameter Abuse (Key ID Injection)
  - [ ] **Description:** Inject paths or payloads within the "kid" parameter in JWT header to force server to use alternate validation key or cause system failures.
  - [ ] **Variants to Test:**
    - **Directory Traversal in "kid":** Change `"kid": "../../../dev/null"` or `"kid": "../../../etc/passwd"` to force backend to validate token signature with empty secret or system file with known content.
    - **SQL Injection in "kid":** If backend searches for key in database using ID, inject SQLi payloads: `"kid": "1' UNION SELECT 'my_secret'--"`.
    - **URL Injection in "kid":** Change "kid" to point to your own HTTP resource returning a specific secret.
  - [ ] **Success Signals:** Successful access with tokens signed by you using empty or predictable secret.
  - [ ] **Tools:** `jwt_tool`, custom Python PoC.

- ### 3.4 Insecure Deserialization / Token Brute Force
  - [ ] **Description:** Attempt to break the symmetric signature secret (HS256) of token through brute force to generate unlimited valid tokens.
  - [ ] **Variants to Test:**
    - Offline brute force of signature using massive password dictionaries.
  - [ ] **Success Signals:** Obtaining original token secret key.
  - [ ] **Tools:** `hashcat` (mode 16500), `john` (`jwt2john.py`).

## ­¤¦▒ 4. Access Control and Business Logic (BOLA / BFLA / Mass Assignment)
*Exploitation of logical failures in calls that allow accessing, modifying, or destroying others' resources.*

- ### 4.1 Broken Object Level Authorization (BOLA / IDOR)
  - [ ] **Description:** Access information or modify other users' resources by manipulating object identifiers present in the request.
  - [ ] **Variants to Test:**
    - Modification of sequential IDs in URI Path: `/api/v1/users/1001` -> `/api/v1/users/1002`.
    - Modification of IDs in JSON request body: `{"user_id": 992}` -> `{"user_id": 991}`.
    - Modification of IDs in cookies or custom headers (`X-User-ID`, `X-Account-ID`).
    - UUID abuse: If API requires UUID, search for public search endpoints returning UUIDs of other users and use them in private calls.
  - [ ] **Success Signals:** Viewing private information of other users or confirmation of unauthorized changes.
  - [ ] **Tools:** Burp Suite Extension *Autorize*, Postman.

- ### 4.2 Broken Function Level Authorization (BFLA)
  - [ ] **Description:** Execute functions or actions reserved for high-privilege users using a low-privilege account.
  - [ ] **Variants to Test:**
    - Access to administrative endpoints deduced logically: `/api/v1/users/me` -> `/api/v1/admin/users`.
    - Change HTTP verbs for blocked functions: regular user can `GET /api/v1/products` but try `DELETE /api/v1/products/25` to delete a product.
    - Send override headers to bypass internal firewalls: `X-Original-URL: /api/v1/admin/settings`.
  - [ ] **Success Signals:** HTTP `200 OK` or `201 Created` executing restricted action, instead of `403 Forbidden`.
  - [ ] **Tools:** Burp Suite.

- ### 4.3 Privilege Escalation via Mass Assignment
  - [ ] **Description:** Force server's data model to process administrative fields or parameters that user shouldn't be able to modify in their profile or creation requests.
  - [ ] **Variants to Test:**
    - During registration (`POST /register`) or profile update (`PUT /users/me`), add privilege fields to JSON: `{"username": "test", "is_admin": true, "role": "admin", "permissions": ["*"]}`.
    - Modify balances or credits in order creation: `{"product_id": 5, "price": 0.01}` or `{"quantity": -10}`.
  - [ ] **Success Signals:** Successful responses reflecting user's role change or confirmation of manipulated order.
  - [ ] **Tools:** Postman, Burp Repeater.

- ### 4.4 Tenant Isolation Invasion (Tenant Breakout)
  - [ ] **Description:** Jump logical boundaries of Multi-Tenant environment to access other customers' or organizations' databases.
  - [ ] **Variants to Test:**
    - Modify `X-Organization-ID` header or parameters like `?orgId=1` in critical calls.
    - Mix valid tokens from Organization A with object identifiers from Organization B.
  - [ ] **Success Signals:** Access or modification of data belonging to another company or organization.
  - [ ] **Tools:** Burp Suite.

- ### 4.5 Business Logic Flaws
  - [ ] **Description:** Break expected application logic flow to obtain illegitimate benefits (e.g., coupon stacking, transfers without funds, etc.).
  - [ ] **Variants to Test:**
    - **Flow Manipulation:** Execute Step 3 (e.g., confirm payment) without completing Step 2 (e.g., process payment).
    - **Use of Negative Values:** Purchases of negative quantities to reduce global account or inject balances.
    - **Race Conditions:** Send multiple coupon redemption requests simultaneously so system processes all before updating "coupon already used" status.
  - [ ] **Success Signals:** Transactions processed for incorrect amounts, screen skipping, or multiple redemptions.
  - [ ] **Tools:** Turbo Intruder (Burp Suite), local Python script for concurrency.

## ­¤Æź 5. Data Injection Attacks (SQLi / NoSQLi / RCE / SSTI)
*Attack vectors where API interprets user data as executable server-side commands.*

- ### 5.1 SQL Injection (SQLi) in Parameters and JSON Payloads
  - [ ] **Description:** Alter relational database queries through API inputs.
  - [ ] **Variants to Test:**
    - Error-based SQLi in search fields.
    - Time-based Blind SQLi within JSON body variables: `{"search": "shoes'; SELECT pg_sleep(5)--"}`.
    - SQL injection in HTTP headers processed and stored by server (e.g., `User-Agent`, `X-Forwarded-For`).
  - [ ] **Success Signals:** Delays in server responses (Time-based), detailed SQL syntax errors in response, or direct record exfiltration.
  - [ ] **Tools:** `sqlmap` (indicating injection point with `*` in Burp request).

- ### 5.2 NoSQL Injection (MongoDB, Redis, etc.)
  - [ ] **Description:** Bypass non-relational database searches, usually implemented in APIs running on Node.js platforms.
  - [ ] **Variants to Test:**
    - Use of logical comparison operators in JSON inputs: `{"username": {"$ne": "guest"}, "password": {"$gt": ""}}`.
    - JavaScript code injection to bypass filters: `{"$where": "this.user == 'admin' || 1==1"}`.
  - [ ] **Success Signals:** Successful login without valid credentials or unexpected database document listing.
  - [ ] **Tools:** Postman, Burp Suite.

- ### 5.3 System Command Injection (Command Injection)
  - [ ] **Description:** Force web server to execute operating system commands (RCE) through parameters passed to console.
  - [ ] **Variants to Test:**
    - Command concatenation with special characters: `filename=image.jpg; id`, `filename=image.jpg | whoami`, `filename=$(id)`.
    - Bypass whitespace filters using `$IFS` or redirection operators.
  - [ ] **Success Signals:** Presence of system command output (`uid=...`, `groups=...`) in HTTP response or reverse network connections (`ping/curl` to your server).
  - [ ] **Tools:** `ffuf` (with command injection dictionary), Burp Suite.

- ### 5.4 Server-Side Template Injection (SSTI)
  - [ ] **Description:** Inject executable code into backend template engines (Jinja2, Twig, Freemarker, Velocity) when API generates dynamic outputs insecurely (emails, PDFs).
  - [ ] **Variants to Test:**
    - Send mathematical strings to identify engine: `{{7*7}}`, `${7*7}`, `<%= 7*7 %>`, `#{7*7}`.
    - RCE exploitation in specific engines (e.g., Jinja2 in Python, Freemarker in Java).
  - [ ] **Success Signals:** Response reflects mathematical result (`49`) or injected command output on server.
  - [ ] **Tools:** Burp Suite, Tplmap.

## ­¤ōä 6. XML Injections and SOAP Attacks (XXE / WSDL / SOAP Abuse)
*Risks associated with legacy or XML-based enterprise architecture APIs.*

- ### 6.1 XML External Entity (XXE) Classic and Blind XXE
  - [ ] **Description:** Force server's XML parser to process references to local or remote external entities.
  - [ ] **Variants to Test:**
    - **LFI via XXE:** Read system files by declaring local SYSTEM entity (e.g., `file:///etc/passwd`).
    - **SSRF via XXE:** Oblige XML parser to perform HTTP requests to cloud metadata or internal services.
    - **Content-Type Confusion:** Change valid JSON request to XML by modifying header to `Content-Type: application/xml` and sending XXE payload.
  - [ ] **Success Signals:** Local file content (e.g., `/etc/passwd`) reflected in server response, or HTTP request received on your controlled server (Blind XXE).
  - [ ] **Tools:** Burp Suite.

- ### 6.2 Exploitation of Exposed WSDL Files and SOAP Action Manipulation
  - [ ] **Description:** Abuse SOAP specifications (WSDL) to discover undocumented methods or inject payloads into SOAP action headers.
  - [ ] **Variants to Test:**
    - Modify `SOAPAction` header in HTTP request to invoke alternate or higher-level functions.
    - Inject XXE or SQLi payloads within `<soap:Envelope>` fields.
  - [ ] **Success Signals:** Server responses with signatures or data from endpoints your user shouldn't access.
  - [ ] **Tools:** SoapUI, Burp Extension *Wsdler*.

## ­¤īÉ 7. Server-Side Request Forgery (SSRF) and URL Abuse
*Force API server to perform arbitrary network requests under attacker control.*

- ### 7.1 SSRF in PDF Generators, Webhooks, and Avatar Upload
  - [ ] **Description:** Abuse functionalities processing user-entered URLs to interact with internal networks or Cloud infrastructure.
  - [ ] **Variants to Test:**
    - **Cloud Metadata IP Access:** `http://169.254.169.254/latest/meta-data/` (AWS), `http://metadata.google.internal/computeMetadata/v1/` (GCP).
    - **Internal Port Scanning:** `http://127.0.0.1:8080`, `http://localhost:3306`, `http://localhost:27017`.
    - **Filter Bypass:** Use decimal IP encodings (`http://2130706433`), dynamic redirect hosts, or IPv6 jumps (`http://[::]:80`).
  - [ ] **Success Signals:** Responses reflecting internal service data (e.g., AWS IAM credentials), notable response time variations indicating open/closed ports, or requests received on your controlled server.
  - [ ] **Tools:** Burp Intruder (for internal port scanning), Burp Collaborator.

## ­¤ōż 8. Insecure File Uploads and Directory Traversal
*Dangers derived from uploading malicious files or manipulating server filesystem.*

- ### 8.1 Insecure File Upload / Extension Filter Evasion
  - [ ] **Description:** Upload executable files (e.g., webshells in `.php`, `.jsp`, `.asp`) to API server simulating image uploads.
  - [ ] **Variants to Test:**
    - **Double Extension Evasion:** `shell.png.php`, `shell.php.jpg`.
    - **Content-Type Evasion:** Upload PHP script modifying header to `Content-Type: image/png`.
    - **Null Character Evasion:** `shell.php%00.png`.
    - **Polyglot Upload:** Files valid as images but containing malicious code executable in metadata.
  - [ ] **Success Signals:** Confirmation of file saved with its public access path, allowing webshell invocation.
  - [ ] **Tools:** Burp Suite.

- ### 8.2 Path Traversal in Filename and Downloads
  - [ ] **Description:** Exit restricted upload or download folders to overwrite or read critical operating system files.
  - [ ] **Variants to Test:**
    - **Directory Traversal in Upload:** Modify filename parameter to `filename="../../../../var/www/html/shell.php"` to overwrite web server root.
    - **Directory Traversal in Download:** Modify resource download path: `/api/v1/download?file=../../../../etc/passwd`.
  - [ ] **Success Signals:** Successful download of operating system file or confirmed remote file overwrite.
  - [ ] **Tools:** Burp Suite, `ffuf`.

## ­¤ĢĖ’ĖÅ 9. GraphQL-Specific Attacks
*Exploitation vectors exclusive to GraphQL API architecture.*

- ### 9.1 Introspection Abuse and Schema Fuzzing
  - [ ] **Description:** Extract complete definition of types, queries, and mutations if introspection isn't disabled.
  - [ ] **Variants to Test:**
    - Send standard introspection query to GraphQL endpoint.
    - If introspection blocked, perform intelligent schema fuzzing based on GraphQL error suggestions (e.g., *"Did you mean...?"*).
  - [ ] **Success Signals:** Obtaining complete list of available mutations and queries (revealing hidden administrative functions).
  - [ ] **Tools:** Burp Extension *InQL*, GraphQL-Voyager.

- ### 9.2 GraphQL Denial of Service (GraphQL DoS) and Query Depth Abuse
  - [ ] **Description:** Collapse server resources by sending massive queries or cyclic nested operations without depth limits.
  - [ ] **Variants to Test:**
    - **Recursive Query:** Request deep recursive relationships (e.g., author -> posts -> author -> posts -> author...).
    - **Batching Abuse:** Send array with hundreds of individual legitimate queries in single HTTP payload to overload backend processor.
  - [ ] **Success Signals:** Extremely high response times or denial of service (HTTP 500 / Timeout) on endpoint.
  - [ ] **Tools:** Burp Suite.

## ­¤īÉ 10. CORS Restriction Evasion and Browser Risks
*Exploit improper configurations in cross-origin resource sharing policies.*

- ### 10.1 Arbitrary Origin Reflection (CORS Origin Reflection)
  - [ ] **Description:** Server accepts and validates any origin making the request, allowing attacker site scripts to steal user's confidential information.
  - [ ] **Variants to Test:**
    - Send HTTP request with `Origin: http://evil.com` header.
    - Send request with `null` origin (`Origin: null`).
    - Validate poor wildcard bypasses (e.g., `http://localhost.target.com`, `http://target.com.evil.com`).
  - [ ] **Success Signals:** Server response includes headers:
    *   `Access-Control-Allow-Origin: http://evil.com` (or injected origin).
    *   `Access-Control-Allow-Credentials: true` (allowing exfiltration of data requiring cookies/session).
  - [ ] **Tools:** Burp Suite.

## ­¤Ü½ 11. Resource Abuse, Rate Limits, and API DoS
*Attacks aimed at exhausting server computational resources or bypassing rate-limit protection.*

- ### 11.1 Rate Limit Bypass and Resource Exhaustion
  - [ ] **Description:** Perform massive requests bypassing protection filters or abusing "expensive" operations on server.
  - [ ] **Variants to Test:**
    - **Rate Limit Bypass:** Change endpoint extension or add characters (e.g., `/api/v1/users` -> `/api/v1/users.json` or `/api/v1/users/`).
    - **Pagination Abuse:** Modify page size parameters to extreme values: `/api/v1/products?limit=9999999` to force database to dump thousands of records to memory in single call.
  - [ ] **Success Signals:** Constant successful HTTP `200` responses to repetitive high-volume requests, or temporary backend outages.
  - [ ] **Tools:** `ffuf`, Burp Intruder.

## ŌÜÖ’ĖÅ 12. Configuration Errors and Hardening (Security Misconfigurations)
*Administration and deployment failures of the API server.*

- ### 12.1 Detailed Error Message Exposure (Verbose Errors)
  - [ ] **Description:** Trigger controlled API failures to obtain error traces, server filesystem paths, or exact technology versions.
  - [ ] **Variants to Test:**
    - Send invalid data types in JSON payloads (e.g., send String where Integer expected, or malformed JSON with unmatched braces).
    - Send empty or null data in mandatory fields.
  - [ ] **Success Signals:** HTTP `500 Internal Server Error` responses returning stack traces, internal server folder paths, or database engine dumps.
  - [ ] **Tools:** Burp Suite.

## ­¤¦¬ 13. Parser, Serialization, and Data Type Abuse
*Exploitation of discrepancies in how frameworks decode and process data structures.*

- ### 13.1 Content-Type Confusion and Insecure Deserialization
  - [ ] **Description:** Force API backend to deserialize malicious objects or decode unexpected data types.
  - [ ] **Variants to Test:**
    - **Content-Type Confusion:** Force server to accept unexpected format (e.g., change `application/json` to `application/yaml` or `application/x-java-serialized-object`).
    - **Type Confusion:** If server expects key-value object, pass array or boolean type to generate unexpected validation behavior.
  - [ ] **Success Signals:** Remote code execution (RCE) via deserialization, or authentication validation bypass.
  - [ ] **Tools:** Ysoserial (for Java deserialization), Burp Suite.

## ­¤ōĪ 14. Transport Channel and Intermediary Attacks (HTTP Request Smuggling)
*Interfere with communication between reverse proxies, load balancers, and backend servers.*

- ### 14.1 HTTP Request Smuggling (CL.TE / TE.CL / TE.TE)
  - [ ] **Description:** Exploit inconsistencies in `Content-Length` (CL) and `Transfer-Encoding` (TE) header processing between front-end server and back-end server.
  - [ ] **Variants to Test:**
    - Send HTTP/1.1 requests with both headers modified so a portion of request is interpreted as start of next user's request in backend.
  - [ ] **Success Signals:** Responses with traces of other lab users' sessions or complete bypass of perimeter proxy filters.
  - [ ] **Tools:** Burp Suite Extension *HTTP Request Smuggler*.
