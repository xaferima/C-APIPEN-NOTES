## 💻 Simulation Scenarios (Mock CTF)

These 8 challenges emulate the vulnerabilities and exploitation steps you must execute in the real `https://$TARGET_HOST` environment to capture the flags.

### Challenge 1: Swagger UI Audit (BFLA)
- **Context:** You discover an `/swagger-ui.html` endpoint using `ffuf` (see commands in [[6. Audit Checklist]]). 
- **Objective:** Find and abuse a hidden administrative function.
- **Action:** 
  1. Review the endpoints listed in the Swagger.
  2. Detect `/api/v1/admin/users/{id}` that accepts `DELETE` methods.
  3. Execute the request with your low-privilege user token. If the user is deleted and the API returns the flag, it's a Broken Function Level Authorization (BFLA).

### Challenge 2: JWT Authentication Evasion
- **Context:** When logging in, the API assigns you a JSON Web Token (JWT) with the claim `"role": "user"`.
- **Objective:** Escalate privileges to `admin` by manipulating the token signature.
- **Action:** 
  1. Capture the token intercepted in Burp Suite.
  2. Modify the header to `{"alg": "none"}` and the payload to `"role": "admin"`.
  3. Remove the signature keeping the final dot intact (e.g. `header.payload.`).
  4. 🔗 *Ref:* [[5. Payloads and Cheat Sheet#6. 🛡️ JSON Web Tokens (JWT)]]

### Challenge 3: Broken Object Level Authorization (BOLA / IDOR)
- **Context:** You have access to the endpoint listing your own receipts at `/api/v1/receipts?id=1052`.
- **Objective:** Access the receipt or invoice of the administrator user (ID `1001`).
- **Action:** 
  1. Try direct access by changing the parameter to `id=1001`.
  2. If direct access is blocked, try HTTP Parameter Pollution (HPP): `?id=1052&id=1001` or pass it as an array `?id[]=1052&id[]=1001`.
  3. 🔗 *Ref:* [[5. Payloads and Cheat Sheet#HPP (HTTP Parameter Pollution)]]

### Challenge 4: Privilege Escalation via Mass Assignment
- **Context:** You can edit your profile by sending a `PUT` to `/api/v1/users/me` with the JSON body: `{"email": "your@email.com", "name": "Your Name"}`.
- **Objective:** Become an administrator of the platform.
- **Action:**
  1. Intercept the update request.
  2. Inject privilege fields in the JSON assuming the backend saves them blindly: `{"email": "your@email.com", "name": "Your Name", "is_admin": true, "role": "admin"}`.
  3. 🔗 *Ref:* [[5. Payloads and Cheat Sheet#Mass Assignment / BOLA]]

### Challenge 5: Data Injection (SQLi in JSON)
- **Context:** There's a product search endpoint `/api/v1/products/search` that receives the payload `{ "query": "shoes" }`.
- **Objective:** Extract information from the database or bypass system authentication.
- **Action:**
  1. Perform fuzzing with structured SQLi payloads to not break JSON format (e.g. `"shoes' OR '1'='1"`).
  2. If the backend uses MongoDB, use injection based on NoSQL operators: `{ "query": {"$ne": null} }`.
  3. 🔗 *Ref:* [[5. Payloads and Cheat Sheet#1. 🗄️ Database Injections]]

### Challenge 6: Server-Side Request Forgery (SSRF)
- **Context:** The API has functionality to import an avatar image from an external URL: `/api/v1/avatar/upload?url=http://...`
- **Objective:** Force the server to access a restricted internal service (e.g. port 80 of the local host).
- **Action:**
  1. Send as parameter the URL `http://127.0.0.1:80` or `http://localhost/admin`.
  2. Analyze if the API response (or response time) reveals access to the internal network and search for the flag.
  3. 🔗 *Ref:* [[5. Payloads and Cheat Sheet#3. 🌐 Server-Side Request Forgery (SSRF)]]

### Challenge 7: Server Side Template Injection (SSTI)
- **Context:** When configuring or updating your username, the system sends an automated email or generates a dynamic PDF that includes the greeting "Hello, [Your Name]".
- **Objective:** Achieve remote code execution (RCE) or extract the flag from the template engine.
- **Action:**
  1. Change your name to the mathematical payload `{{7*7}}` or `${7*7}`.
  2. If the PDF or message renders "Hello, 49", the SSTI attack is successful.
  3. 🔗 *Ref:* [[5. Payloads and Cheat Sheet#4. 🧩 Server Side Template Injection (SSTI)]]

### Challenge 8: Insecure File Upload & Directory Traversal
- **Context:** The endpoint `/api/v1/files/upload` allows uploading images (PNG/JPG).
- **Objective:** Upload an executable script or overwrite a sensitive file on the server.
- **Action:**
  1. Intercept the valid file upload request.
  2. Change the `filename` field by injecting directory skip sequences: `filename="../../../var/www/html/shell.php"`.
  3. If there's a filter, try bypassing extensions using double extensions (`shell.png.php`) or altering the Content-Type.

### Challenge 9: XML External Entity (XXE) in REST/SOAP APIs
- **Context:** You find a SOAP service with its `.wsdl` definition or a REST endpoint that accepts input data (though it expects JSON).
- **Objective:** Exfiltrate a critical file from the server, such as `/etc/passwd`.
- **Action:**
  1. Capture the original request.
  2. If the endpoint expects JSON, modify the `Content-Type` header to `application/xml` or `text/xml`.
  3. Inject the XXE payload by defining the external entity in the DOCTYPE declaration.
   4. 🔗 *Ref:* [[5. Payloads and Cheat Sheet#2. 📄 XML External Entity (XXE)]]

### Challenge 10: Command Injection (Command Injection)
- **Context:** A `/api/v1/network/ping` endpoint is found, for example, that accepts an `ip` parameter or an image processing service.
- **Objective:** Achieve remote command execution on the server.
- **Action:**
  1. Try concatenating shell commands using metacharacters: `;`, `|`, `&&`, or `` ` `` (backticks). Example: `ip=127.0.0.1; id`.
  2. If spaces are filtered, use bypass techniques like `$IFS` or redirections.
   3. 🔗 *Ref:* [[5. Payloads and Cheat Sheet#5. ⚙️ Command Injection (Command Injection)]]

### Challenge 11: Brute Force Attacks and Rate Limiting
- **Context:** A login endpoint (`/api/v1/login`) doesn't seem to have obvious lockouts after several failed attempts.
- **Objective:** Execute a password spraying attack without being blocked.
- **Action:**
  1. Send brute force requests using `ffuf` or Intruder.
  2. If a block (HTTP 429) appears, try injecting headers to trick IP tracking: `X-Forwarded-For: 127.0.0.1`, `X-Originating-IP: 127.0.0.1`.
  3. If it's a JSON, try grouping multiple credentials in a single request (batching).

### Challenge 12: Introspection and Mutations Attacks in GraphQL
- **Context:** You discover a `/graphql` endpoint using your fuzzing script.
- **Objective:** Find hidden mutations and escalate privileges.
- **Action:**
  1. Send the universal introspection query to map the entire schema (queries and mutations).
  2. Identify a mutation like `updateUserRole`.
  3. Use GraphQL-Voyager or InQL to structure the request and send it to convert the user to administrator.
   4. 🔗 *Ref:* [[5. Payloads and Cheat Sheet#GraphQL Introspection (Discover schemas)]]

### Challenge 13: CORS Restriction Bypass
- **Context:** A sensitive data endpoint (e.g. `/api/v1/user/keys`) seems available if an authenticated user visits it.
- **Objective:** Demonstrate that an external attacker site can steal that data.
- **Action:**
  1. Intercept the request and inject the `Origin: http://evil.com` header.
  2. Observe if the response reflects the `Access-Control-Allow-Origin: http://evil.com` header along with `Access-Control-Allow-Credentials: true`.
  3. In this scenario, the flag represents the successful theft of data through the misconfigured CORS policy.

### Challenge 14: Business Logic (Manipulation of Negative/Quantity Values)
- **Context:** In a simulated online store or transfers API `/api/v1/transfer`.
- **Objective:** Take advantage of the lack of type validation to cause anomalous behavior.
- **Action:**
  1. Modify the `amount` parameter of the transfer to a negative value (e.g. `{"amount": -5000}`).
  2. If the balance increases, or if the parameter allows very long strings causing a crash, you've successfully exploited a Business Logic flaw.

### Challenge 15: Password Reset Attacks (Host Header Injection)
- **Context:** A `/api/v1/password-reset` endpoint processes requests by sending an email to the user.
- **Objective:** Hijack the reset token link.
- **Action:**
  1. Intercept the password reset request from the administrator's account.
  2. Modify the HTTP `Host: INSERTHOST.COM` header to `Host: attacker.com`.
  3. If the API generates the link using this header and you capture it, you've solved the challenge.
