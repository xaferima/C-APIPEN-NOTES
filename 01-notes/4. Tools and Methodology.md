# Tools and Methodology (Deep Dive)

To pass the C-APIPen in 4 hours, you need a well-configured set of tools and a systematic testing methodology. Without AI available during the actual exam, speed in your manual workflow will be key.

## 🛠️ Required Tools (Arsenal)

### 1. Main Interception and Analysis
- **Burp Suite (Community / Pro):** It's your central tool. Make sure you have useful extensions configured:
  - *OpenAPI Parser / Swagger Parser:* To parse and import Swagger JSON definitions and generate requests.
  - *JSON Web Tokens (JWT):* To analyze signatures, inject "none" algorithm, and attempt key confusion attacks.
  - *Autorize:* Very useful for finding BOLA / IDOR (testing requests with cookies from different users in parallel).
  - *GraphQL Raider / InQL:* To work comfortably with GraphQL (introspection, variables, etc).
- **Postman:** Fundamental for organizing collections (`.json`). The syllabus explicitly mentions "Import and Manage API Collections in Postman". Learn to use Postman anchored to Burp's proxy (setting the local proxy in Postman to `127.0.0.1:8080`) so all traffic is saved in HTTP History.
  * **Proxy Configuration:** Go to *Settings* -> *Proxy*, enable *Add a custom proxy configuration* (HTTP/HTTPS) pointing to `127.0.0.1:8080`. Disable *Use System Proxy*.
  * **Burp's CA Certificate:** To avoid SSL errors (`Self-signed certificate in chain`), visit `http://burp` from your browser with the proxy active, download `cacert.der` and import it in Postman (*Settings* -> *Certificates* -> *CA Certificates*). Alternatively, you can disable *SSL certificate verification* in *Settings* -> *General*.

### 2. Discovery and Enumeration (Fuzzing)
- **Kiterunner:** Specific tool for discovering API endpoints that conventional tools like Dirbuster don't detect due to lacking HTTP verbs or specific parameters.
  * **Usage workflow:**
    ```bash
    # Basic API scan mapping traditional verbs
    kr brute https://mock.hackme.secops.group:9000/ -w ~/Tools/SecLists/Discovery/Web-Content/api/api-endpoints.txt
    
    # Scan using precompiled routes database (.kite)
    kr scan https://$TARGET_HOST -A=apiroutes-260227
    ```
- **Arjun:** Tool dedicated exclusively to discovering hidden GET, POST, and JSON parameters that aren't explicit in the Swagger.
  * **Usage workflow:**
    ```bash
    # Find hidden GET parameters in a known endpoint
     arjun -u https://$TARGET_HOST/api/v1/users -m GET -w ~/Tools/SecLists/Discovery/Web-Content/burp-parameter-names.txt
    
    # Find parameters in JSON bodies (POST)
     arjun -u https://$TARGET_HOST/api/v1/login -m POST --headers "Content-Type: application/json" -w ~/Tools/SecLists/Discovery/Web-Content/burp-parameter-names.txt
    ```
- **ffuf / Wfuzz / Feroxbuster:** To discover hidden directories, fuzz GET/POST parameters, and inject payloads in headers.

### 3. Specific Exploitation
- **SQLMap:** To corroborate or exploit SQL Injections if you find a vulnerable parameter.
- **jwt_tool (ticarpi):** Essential Python script if Burp falls short modifying JWTs (alg=None attacks, kid injection, secret brute-force).
- **GraphQL-Voyager:** If you manage to extract an introspection, Voyager renders a visual map of the entire schema, making it very easy to search for interesting mutations.

## 🔬 Methodology: What to Look For Step by Step (Playbook)

### Phase 1: Reconnaissance and Import (0 - 30 min)
1. **Search for documentation:** Look for `swagger.json`, `/api/v1/docs`, `/api/openapi.json`, `/v1/api-docs`, `/graphql`, or `?wsdl` files (if SOAP).
2. **Import to Postman/Burp:** Load the endpoints. Try to authenticate. Map all roles given to you in the exam (Do you have admin? Regular user? Unauthenticated user?).
3. **Pass everything through the Proxy:** Navigate and execute all requests in Postman through Burp to populate the Site Map.

### Phase 2: Auth & Session (JWT & Reset) (30 - 60 min)
1. **Evaluate Login:** Try basic SQLi (`admin' or '1'='1--`, etc.) or NoSQLi (`{"$gt": ""}`) on authentication endpoints.
2. **Token Review (JWT):** If they return a JWT, put it in *jwt.io* or *jwt_tool*. Try removing the signature (alg=none), try modifying the "role" in the payload, and try sending it.
3. **Brute Force & Rate Limit:** The syllabus mentions "Brute force attacks". Try brute forcing the login endpoint (Hydra/Intruder). Verify if there's an Account Lockout or Rate Limit (HTTP 429 Code). If there's rate limit, try CORS bypasses or inject `X-Forwarded-For: 127.0.0.1`.
4. **Password Reset:** Check the password forgotten endpoint. Can you request a reset for admin and inject the Host header so the link comes to you? (Host Header Injection -> SSRF / Token Leak).

### Phase 3: Authorization (BOLA / IDOR / BFLA) (60 - 120 min)
*APIs suffer from BOLA (Broken Object Level Authorization) more than anything else.*
1. Take the token from your "User A" (normal).
2. Repeat a request that asks for data from another user (e.g: `/api/v1/users/2` or `/api/v1/orders/102`).
3. Observe if it returns the data (IDOR).
4. Try accessing administrative endpoints (`/api/v1/admin/users`) with your regular user token. If you manage to POST to a creation endpoint assuming your user "shouldn't see it", you've found BFLA (Broken Function Level Authorization).

### Phase 4: Data Injection (Mass Assignment, SSTI, XXE, SQLi) (120 - 180 min)
1. **Mass Assignment:** If there's a `/api/v1/users/me` endpoint to update your profile (e.g: change name). Try adding `"role": "admin"`, `"is_admin": true`, or `"account_balance": 9999` in the sent JSON. Poorly configured APIs update fields without validating them.
2. **XXE:** If an endpoint accepts `Content-Type: application/xml` (or is a SOAP service with WSDL), try injecting a classic XXE payload calling `file:///etc/passwd`. Remember that sometimes a JSON endpoint also parses XML if you manually change the Content-Type.
3. **SSTI (Server-Side Template Injection):** If there's any endpoint that returns rendered text (e.g. receipts, emails, profile names on screen), inject `{{7*7}}` or `${7*7}` to see if it responds with `49`.
4. **SSRF:** If a parameter requests a URL (e.g: `avatar_url=http://...`), point to a server you control (Burp Collaborator) to validate if the server makes external requests, or point to `http://localhost/admin` to see if it bypasses filters.

### Phase 5: Protocol-Specific Review
- **GraphQL:** Launch an introspection. Search for hidden mutations (e.g. `updateRole`, `deleteUser`). 
- **SOAP:** Analyze the WSDL by importing it to Burp (Wsdler extension). Inject SQL commands or XXE in `<soap:Envelope>` wrappers.

## 📌 Exam Tips
- **Don't get stuck:** If an endpoint doesn't yield, move to the next. The 4 hours fly by.
- **Take concurrent notes:** Keep a notepad ready with the flags as you get them.
- **Keep Burp clean:** Use scopes in Burp to not fill yourself with background noise, focus only on the exam IP/Hostname.
