# Technical Audit Checklist - C-APIPen

This interactive checklist is structured chronologically according to a 4-hour attack flow to systematically cover the 20 topics in your exam.

---

## 🔍 Phase 1: Reconnaissance, Mapping and Discovery (0:00 - 0:45)

### 1. Interaction with API Definitions (Swagger / OpenAPI / WSDL)
- [ ] **Search for exposed specification files:**
  - Test common endpoints: `/swagger.json`, `/swagger-ui.html`, `/api-docs`, `/v1/api-docs`, `/v2/api-docs`, `/openapi.json`, `/api/v1/swagger.json`.
  - [ ] *Tool:* Use Burp *Swagger Parser* extension or import the endpoint directly in Postman.
- [ ] **Search for SOAP services (WSDL):**
  - Look for `?wsdl` suffixes or endpoints like `/ws/`, `/services/`, `/soap/`.
  - [ ] *Tool:* Import the `.wsdl` in Burp *Wsdler* extension or in *SoapUI* to parse XML actions.
- [ ] **GraphQL Introspection:**
  - Send an introspection query to the `/graphql`, `/api/graphql`, `/graphql-api` endpoint. 
  - 🔗 *Ref:* [[5. Payloads and Cheat Sheet#GraphQL Introspection (Discover schemas)]]
  - [ ] *Tool:* Use *InQL* in Burp to enable/disable the query and map queries and mutations.

### 2. Collection Management and Import (Postman & Burp)
- [ ] **Configure Proxy in Postman:**
  - [ ] Configure Postman to use `127.0.0.1:8080` (Burp Suite proxy) to record each test request.
  - [ ] Import the discovered or obtained collections.
  - [ ] Fill in global environment variables in Postman (Base URL, authentication tokens).

### 3. Active Enumeration and Endpoint Fuzzing (Terminal Commands)
*We will use `ffuf` to map resources automatically. Adjust the host variable (`https://$TARGET_HOST`) and dictionary paths.*

- [ ] **Directory and Hidden Endpoint Fuzzing:**
  ```bash
  # Classic path fuzzing using API-specific dictionary
  ffuf -u https://$TARGET_HOST/FUZZ -w ~/Tools/SecLists/Discovery/Web-Content/api/api-endpoints.txt -mc 200,201,202,301,302,401,403 -c -v
  ```
- [ ] **Fuzzing Hidden GET Parameters:**
  ```bash
  # Search for invisible parameters in known URLs (e.g. ?debug=, ?admin=)
  ffuf -u https://$TARGET_HOST/api/v1/users?FUZZ=1 -w ~/Tools/SecLists/Discovery/Web-Content/burp-parameter-names.txt -mc 200 -c -v
  ```
- [ ] **Fuzzing HTTP Verbs/Methods:**
  ```bash
  # Identify accepted verbs (POST, PUT, DELETE, PATCH, etc.) on a specific endpoint
  ffuf -u https://$TARGET_HOST/api/v1/users -X FUZZ -w ~/Tools/SecLists/Fuzzing/http-request-methods.txt -mc 200,201,202,204,401,403 -c -v
  ```

## 🔐 Phase 2: Authentication, Session and CORS (0:45 - 1:30)

### 4. Authentication Flows, Brute Force and Password Spraying
- [ ] **Evaluate Rate Limit on Login (Fuzzing / Brute Force):**
  ```bash
  # Brute force against a JSON body on the login endpoint
  ffuf -u https://$TARGET_HOST/api/v1/login -X POST -H "Content-Type: application/json" -d '{"username": "admin", "password": "FUZZ"}' -w ~/Tools/SecLists/Passwords/Leaked-Databases/rockyou.txt -mc 200,302 -c -v
  ```
  - [ ] Does the API block the account? Does it return HTTP `429 Too Many Requests`?
  - [ ] If it returns an error, try bypassing it using headers: `X-Forwarded-For`, `X-Originating-IP`, `X-Real-IP`, `CF-Connecting-IP`.

### 5. Password Reset Attacks
- [ ] **Host Header Manipulation:**
  - Send a reset request and alter the `Host: attacker.com` header. Verify if the generated token is sent to the attacker's domain.
- [ ] **Email Parameter Manipulation:**
  - Try changing the body from `{"email": "victim@deuna.com"}` by using arrays or multiple parameters: `{"email": ["victim@deuna.com", "attacker@deuna.com"]}` or duplicate parameters in the URL (HTTP Parameter Pollution).
  - 🔗 *Ref:* [[5. Payloads and Cheat Sheet#HPP (HTTP Parameter Pollution)]]
- [ ] **Brute force on recovery token:**
  ```bash
  # If the token sent by email is a 4-digit PIN code
  ffuf -u https://$TARGET_HOST/api/v1/reset-password -X POST -H "Content-Type: application/json" -d '{"email": "admin@target.com", "pin": "FUZZ", "new_password": "Password123!"}' -w <(seq -w 0000 9999) -mc 200 -c -v
  ```

### 6. Attacks on JSON Web Tokens (JWT)
- [ ] **Test for None Algorithm (`alg=none`):**
  - Modify the JWT header to `{"alg": "none"}` or variants (`None`, `NONE`, `nOnE`), remove the JWT signature (keeping the final dot) and send it.
  - 🔗 *Ref:* [[5. Payloads and Cheat Sheet#6. 🛡️ JSON Web Tokens (JWT)]]
- [ ] **Algorithm Confusion (Key Confusion / RS256 vs HS256):**
  - Change the asymmetric algorithm `RS256` to symmetric `HS256`. Sign the token using the server's public key (if exposed in `/jwks.json` or similar) as the symmetric secret key.
- [ ] **Key Hijacking / Injection (kid):**
  - [ ] Inject Directory Traversal in the `"kid": "../../../dev/null"` header and try signing with an empty secret (`0x00` or empty string).
  - [ ] Inject a URL under your control in `"kid": "http://attacker.com/key"`.
- [ ] **Weak Signature (Brute Force):**
  - Run `jwt_tool -b token.jwt` to try breaking the secret via brute force if a symmetric algorithm (HS256) is used.

### 7. CORS Evasion (Cross-Origin Resource Sharing)
- [ ] **Origin Header Reflection:**
  - Send the request with an `Origin: http://evil.com` header. If the response includes `Access-Control-Allow-Origin: http://evil.com` and `Access-Control-Allow-Credentials: true`, it's vulnerable.
- [ ] **CORS Bypass with wildcards:**
  - Test misconfigured origins: `http://localhost.evil.com`, `http://evil.com.target.com`, `null`.

## 🧱 Phase 3: Access Control and Business Logic (1:30 - 2:30)

### 8. Insecure Direct Object References (IDOR / BOLA)
- [ ] **Verify sequential IDs:**
  - If an API calls an endpoint via `/api/v1/orders/1001`, try changing it to `1002`, `1000`, etc.
- [ ] **Verify non-guessable UUIDs/IDs:**
  - Try to find UUIDs from other users via public endpoints (like user directories `/api/users/search`) and then use them in private endpoints.
- [ ] **Test with two roles (Token A vs Token B):**
  - [ ] *Tool:* Load both users' cookies/tokens in Burp's *Autorize* extension to automate IDOR testing across the entire Site Map passively.

### 9. Parameter Manipulation and Business Logic (BFLA)
- [ ] **Alteration of prices/values in transactions:**
  - Send negative, empty or extremely large values in parameters like `{"amount": -100}`, `{"quantity": 0.5}`, or `"discount": 100}`.
- [ ] **Injection of administrative parameters (Mass Assignment):**
  - During registration or profile editing (`PUT /users/me`), add privileged fields: `{"is_admin": true}`, `{"role": "administrator"}`, `{"permissions": ["*"]}`.
  - 🔗 *Ref:* [[5. Payloads and Cheat Sheet#Mass Assignment / BOLA]]

## 💥 Phase 4: Data, File and Service Injection (2:30 - 3:45)

### 10. Classic Injection (SQLi, NoSQLi, Command Injection)
- [ ] **SQL Injection in JSON Payload (Fuzzing):**
  ```bash
  # Detect blind or error-based SQL injections within a JSON field
  ffuf -u https://$TARGET_HOST/api/v1/products -X POST -H "Content-Type: application/json" -d '{"search": "FUZZ"}' -w ~/Tools/SecLists/Fuzzing/SQLi/Quick-SQLi.txt -mr "SQL syntax|mysql|sqlite|postgresql" -c -v
  ```
  - 🔗 *Ref:* [[5. Payloads and Cheat Sheet#SQL Injection (SQLi)]]
- [ ] **NoSQL Injection:**
  - In JSON search or login inputs, inject MongoDB comparison operators: `{"$gt": ""}`, `{"$ne": null}`.
  - 🔗 *Ref:* [[5. Payloads and Cheat Sheet#NoSQL Injection (MongoDB)]]
- [ ] **Command Injection (RCE Fuzzing):**
  ```bash
  # Inject system commands (e.g. endpoints that process images or files)
  ffuf -u https://$TARGET_HOST/api/v1/compress -X POST -H "Content-Type: application/json" -d '{"filename": "test.jpgFUZZ"}' -w ~/Tools/SecLists/Fuzzing/command-injection-space.txt -mr "uid=|gid=|groups=" -c -v
  ```
  - 🔗 *Ref:* [[5. Payloads and Cheat Sheet#5. ⚙️ Command Injection (Command Injection)]]

### 11. XML External Entity (XXE) and XML Injection
- [ ] **Test XML Parser (REST / SOAP):**
  - Inject classic XXE payloads to try reading system files (`/etc/passwd` or `C:\Windows\win.ini`).
  - If the backend accepts XML but doesn't respond with data (Blind XXE), try injecting an external entity pointing to a server you control (Burp Collaborator) to force SSRF.
  - 🔗 *Ref:* [[5. Payloads and Cheat Sheet#2. 📄 XML External Entity (XXE)]]

### 12. Server Side Template Injection (SSTI)
- [ ] **Injection in dynamic fields:**
  - Test basic detection strings: `{{7*7}}`, `${7*7}`, `<%= 7*7 %>` in profiles, messages, report generation and API emails.
  - Look for whether the API responds with the mathematical execution (`49`).
  - 🔗 *Ref:* [[5. Payloads and Cheat Sheet#4. 🧩 Server Side Template Injection (SSTI)]]

### 13. Server-Side Request Forgery (SSRF)
- [ ] **Audit URL parameters:**
  - Look for parameters like `url=`, `redirect=`, `image=`, `file_path=`.
  - Try calling the server's local port `http://127.0.0.1:80` or cloud services (`http://169.254.169.254/`).
  - 🔗 *Ref:* [[5. Payloads and Cheat Sheet#3. 🌐 Server-Side Request Forgery (SSRF)]]

### 14. Insecure File Uploads
- [ ] **Extension evasion:**
  - Try uploading malicious files (e.g. php, asp, jsp scripts) by modifying the extension (`.phtml`, `.php5`, `.jpg.php`, `.png.jsp`) or bypassing Content-Type filters by changing it to `image/jpeg`.
- [ ] **Directory Traversal in filename:**
  - Change the filename in the request to: `../../../../var/www/html/shell.php` to try overwriting accessible server directories.

### 15. Directory Traversal / File Inclusion
- [ ] **Access to relative paths:**
  - In file download or path parameters (`?file=image.png`), inject directory skip sequences: `../../../../etc/passwd` or using URL encoding (`%2e%2e%2f`).

### 16. GraphQL Attacks (Specific attacks)
- [ ] **Abuse of Intricate Query (DoS):**
  - Design recursive nested queries if there's no depth limit in GraphQL.
- [ ] **Query bypass:**
  - Try searching for dangerous mutations like `registerAdmin` or modify query parameters to gain unauthorized access.

---

## 🧹 Phase 5: Misconfigurations and Closure (3:45 - 4:15)

### 17. Security Misconfigurations
- [ ] **Check for Verbose Error Messages:**
  - Send requests with invalid payloads (e.g. broken JSON) to force stack-traces, database dumps, or information about specific framework and system versions.
- [ ] **Bypass of unauthorized HTTP Methods:**
  - If `GET` is blocked, try `HEAD`, `OPTIONS`, `PUT` or use method override headers like `X-HTTP-Method-Override: PUT`.

### 18. Rate Limiting and Resource Abuse
- [ ] **Bypass rate limit rules:**
  - If an endpoint is blocked by rate limit, check if you can bypass the rule by changing the extension or adding extra characters: `/api/v1/users` -> `/api/v1/users.json` or `/api/v1/users/`.
